# Admissions Interview Preparation

## 1. What problem does your application solve?

This application is built for anyone dealing with mental-health challenges. After a difficult moment, they may not remember exactly how they felt or what may have triggered it. Without those details, it can be harder to process the experience and work through it with a therapist.

## 2. What value does your application create for the user?

The application gives users a place to record their personal feelings and what has upset them while the experience is still fresh. Later, they can revisit that emotional experience whenever they want. If they choose to share the information with their therapist, it may help them discuss what happened, recognize possible patterns, and explore helpful coping strategies together.

## 3. Why did you choose this solution?

I chose a digital diary because journaling gives people a space for personal reflection. Making it digital allows users to record an experience while it is still fresh and revisit it later. I designed the application to look and feel like a personal diary because that familiarity can create a sense of comfort and ownership, while recognizing that the visual design is separate from how the user's information is stored and protected.

## 4. How did you decide what to build first?

I identified the smallest version of the application that would prove its value: a user must be able to write an entry, save it, and revisit it later. I decided to build and test that core functionality before adding personalization, styling, images, and animation. This approach makes the code easier to understand and allows me to debug the essential features before introducing more complex visual elements.

## 6. Was there a time you questioned or challenged an AI suggestion?

I originally planned to build the diary with React, Express, and Neon, but an AI review showed that this conflicted with the admissions requirement to deploy a static HTML, CSS, and JavaScript application through GitHub Pages. The static recommendation made sense, but I questioned whether `localStorage` could adequately protect personal entries by adding a username and password.

Instead of assuming that suggestion was secure, I asked AI to verify it and learned that a JavaScript login screen would not provide real authentication because JavaScript running on the website can still access the stored data. I decided to treat this submission as a functional prototype using non-sensitive demonstration entries, clearly document its limitations, and identify authenticated, per-user storage as a future production feature.

This experience showed me that I should challenge AI recommendations, verify important claims, and make the final decision myself.
